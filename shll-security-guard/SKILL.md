# SHLL Security Guard

Pre-execution security validation layer for AI agent on-chain transactions on BNB Chain (BSC).

## What This Skill Does

This skill validates transaction payloads against SHLL's on-chain PolicyGuard **before** they are executed. It does NOT execute trades itself. It is designed to work alongside trading skills (swap, bridge, lend) as a security checkpoint.

**Workflow with other skills:**

```
[Any Trading Skill] → generates tx payload
        ↓
[SHLL Security Guard] → validates against 6 policy layers
        ↓
    ✅ PASS → safe to sign and broadcast
    ❌ REJECT → blocks execution, returns reason
```

## Why Use This

AI agents using trading skills (DEX swap, bridge, lending) have no built-in protection against:
- **Model hallucinations**: LLM generates incorrect parameters (wrong token, extreme slippage)
- **Prompt injection**: Malicious data sources trick agents into unauthorized transfers
- **Overspending**: No per-action or daily spending caps
- **Rug pulls**: Interaction with unverified token contracts

SHLL Security Guard enforces **on-chain, contract-level** validation that the AI itself cannot bypass or modify.

## Security Architecture

SHLL uses a dual-wallet model:

- **Owner wallet** (user-controlled): Holds ownership rights, manages high-risk admin operations
- **Operator wallet** (AI-controlled): Restricted execution rights only, cannot transfer funds outside the vault

All agent funds are held in **AgentAccount** vaults (ERC-6551 Token Bound Accounts), isolated from the operator wallet. PolicyGuard validates every action before the vault releases funds.

### Six-Layer Policy Matrix

| Layer | Policy | What It Blocks |
|---|---|---|
| 1 | **SpendingLimit** | Transactions exceeding per-action or daily caps |
| 2 | **CooldownPolicy** | Rapid-fire transactions (bot loops, gas drain) |
| 3 | **ReceiverGuard** | Transfers to non-whitelisted addresses |
| 4 | **DeFiGuard** | Calls to non-approved contract functions |
| 5 | **DexWhitelist** | Routing through unverified DEX contracts |
| 6 | **TokenWhitelist** | Swaps involving non-approved tokens |

All policies are enforced at the smart contract level. The AI agent has no capability to disable or bypass them.

## Usage

### Step 1: Check if agent has a SHLL vault

```bash
shll-run status -k <tokenId>
```

If the user has no token ID, use the onboarding flow:
```bash
shll-run setup-guide
```

### Step 2: Validate a transaction (read-only, no gas)

```bash
shll-run policies -k <tokenId>
```

This returns all active policies and their current parameters for the agent vault.

### Step 3: Execute with policy enforcement

When executing any DeFi operation, SHLL automatically validates against all active policies:

```bash
# Swap with built-in validation
shll-run swap -f BNB -t USDT -a 0.1 -k <tokenId>

# Check portfolio
shll-run portfolio -k <tokenId>
```

### Step 4: Cross-skill execution (MCP)

For transaction payloads generated by other skills/aggregators:

```
MCP Tool: execute_calldata
  - tokenId: <agentTokenId>
  - target: <contract address>
  - calldata: <hex payload from other skill>
```

PolicyGuard will decode and validate the calldata before execution. If the recipient, token, or function selector violates any policy, the transaction is rejected with a specific reason.

## Integration with Other Skills

SHLL Security Guard adds a validation layer to any on-chain trading workflow:

| Combined With | Workflow |
|---|---|
| DEX swap skills | Validate token + DEX whitelist before swap execution |
| Bridge skills | Validate receiver address + spending limit before bridging |
| Lending skills | Validate protocol whitelist + function selector before deposit |
| NFT trading skills | Validate contract address + spending limit before purchase |
